## User

Section "Tunnel Loop (Transport Packets)" from `.anatoly/docs/02-Architecture/03-Data-Flow.md`:

The tunnel loop is the steady-state hot path. Once a session is established, plaintext IP packets enter via a TUN device and exit as encrypted UDP datagrams — and vice versa.

### Outbound (TUN → UDP)

1. **rustguard-tun** reads a plaintext IP packet from the TUN device (Linux /dev/net/tun, macOS utun, or AF_XDP zero-copy path).
2. **rustguard-core** looks up the session by destination IP via the AllowedIPs table. It selects the current send key and atomically increments the 64-bit nonce counter.
3. **rustguard-crypto** encrypts the payload with ChaCha20-Poly1305. The nonce is derived from the counter. The 16-byte authentication tag is appended.
4. **rustguard-core** prepends the 4-byte type field and 4-byte receiver index to produce the final wire frame, which is sent over UDP.

### Inbound (UDP → TUN)

1. **rustguard-core** receives a UDP datagram, extracts the receiver index, and locates the session.
2. The **replay window** check() is called before any decryption. Packets with a counter already seen — or below the window floor — are dropped immediately.
3. **rustguard-crypto** performs ChaCha20-Poly1305 AEAD decryption. Failure aborts; the replay window is *not* advanced.
4. Only after successful decryption does **rustguard-core** call replay window update(), advancing the 2048-bit sliding bitmap.
5. The source IP is verified against AllowedIPs. Valid packets are written to the TUN device by **rustguard-tun**.
