## User

Section "ADR-006 — Replay Window Split into `check()` / `update()`" from `.anatoly/docs/02-Architecture/04-Design-Decisions.md`:

**Status:** Accepted (security fix)

**Context:** The original replay protection implementation advanced the 2048-bit sliding window bitmap before verifying the AEAD tag. A network attacker could replay or corrupt packets with arbitrary counter values, poisoning the window and causing legitimate packets to be dropped.

**Decision:** The replay window exposes two separate operations: check() tests whether a counter would be accepted (without mutating state), and update() advances the window (called only after successful AEAD verification).

**Consequences:**
- Replay window state is never modified by a packet that fails authentication.
- The split API makes the correct ordering impossible to accidentally invert in future call sites.
