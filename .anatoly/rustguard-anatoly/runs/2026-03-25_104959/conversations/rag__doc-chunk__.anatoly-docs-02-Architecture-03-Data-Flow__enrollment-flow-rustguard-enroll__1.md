## User

Section "Enrollment Flow (rustguard-enroll)" from `.anatoly/docs/02-Architecture/03-Data-Flow.md`:

The enrollment flow handles zero-config peer provisioning via the serve / join commands. It is independent of the tunnel loop.

1. **rustguard serve** initialises rustguard-enroll with a CIDR pool. The server holds .1; clients receive sequential IPs from the remainder of the pool.
2. Enrollment is **closed by default**. rustguard open  writes an atomic expiry timestamp over the UNIX domain control socket. rustguard close clears it immediately.
3. When a client calls rustguard join, rustguard-enroll derives an XChaCha20-Poly1305 key from the shared token and uses it to encrypt the key exchange. No plaintext key material crosses the wire.
4. On success, the new peer (public key + assigned IP) is added to rustguard-core's peer table and persisted to ~/.rustguard/state.json so it survives daemon restarts.
