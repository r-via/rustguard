## User

Section "Zero-Config Enrollment" from `.anatoly/docs/02-Architecture/02-Core-Concepts.md`:

### Enrollment Protocol

A custom key-exchange protocol implemented in rustguard-enroll. A token-derived XChaCha20 key encrypts the key exchange, so no TLS or PKI is required. The server allocates an IP from the pool; the client receives its address and peer configuration in a single round trip.

### IP Pool

A CIDR block (--pool 10.150.0.0/24) managed by the enrollment server. The server claims .1; each joining client receives the next unallocated address sequentially.

### Pairing Window

Enrollment is not always-open. A physical-presence model gates enrollment: the server starts with enrollment closed and requires an explicit rustguard open  command to accept new peers. The window auto-closes on expiry via an atomic timestamp. Modelled after Zigbee's pairing concept.

### Control Socket

A UNIX domain socket used to send runtime management commands (open, close, status) to the running daemon without restarting the process.

### Persistence

Server state — peer public keys and their assigned IPs — is serialized to ~/.rustguard/state.json. The server reloads this state on restart; existing peers reconnect without re-enrolling.
